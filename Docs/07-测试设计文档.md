# 智能在线编程练习平台 - 测试设计文档

## 测试设计

### 智能在线编程练习平台测试计划和测试程序

## 1. 概述

### 1.1 系统需求概述

#### 1.1.1 项目背景
在计算机专业教学、求职笔试面试、算法竞赛训练等核心场景中,算法编程能力是衡量开发者技术水平的关键指标。随着编程学习需求的持续增长,传统算法题库系统逐渐暴露出诸多痛点:功能单一,仅支持基础的题目展示与结果判定;评测机制僵化,仅验证代码正确性,无法对代码质量进行深度分析;题目资源固定固化,无法满足个性化练习需求;缺乏专业的指导能力,学习者难以自主优化代码、提升编程规范。

#### 1.1.2 系统定位
智能在线编程练习平台是一款基于 **SpringBoot + Vue 3** 开发的Web端一站式算法学习平台,整合了**标准化题库管理、在线代码编辑、自动化代码评测、AI智能辅助、题目动态生成**五大核心能力。平台采用「传统OJ判题+AI智能赋能」的双引擎模式,为用户提供高效、智能的算法学习解决方案。

#### 1.1.3 核心功能需求
1. **用户认证管理**: 用户注册、登录、退出、权限控制(JWT Token)
2. **用户个人中心**: 信息管理、头像上传、密码修改、提交记录查询、刷题统计
3. **题库管理**: 题目分类、题目列表、题目详情、题解浏览、难度筛选、关键词搜索
4. **在线编程与评测**: 代码编辑、代码提交、自动化评测(AC/WA/CE/RE/TL)、测试用例比对
5. **AI智能模块**: AI题目生成、代码多维度评价(时间复杂度、空间复杂度、代码风格、可读性、优化建议)
6. **管理员后台**: 用户管理、题目管理、分类管理、测试用例管理、题解管理
7. **数据可视化**: 学习数据统计、正确率分析、历史记录回溯

#### 1.1.4 技术需求
- **后端**: Spring Boot 2.7.18, Java 17, MyBatis, MySQL 8.0+
- **前端**: Vue 3, TypeScript 5.x, Vite 4.x, Pinia, Element Plus
- **评测引擎**: Docker沙箱隔离,支持C/C++/Java/Python多语言
- **AI集成**: 大模型API接入,智能题目生成与代码分析
- **文件存储**: 阿里云OSS(头像、测试用例文件)
- **安全机制**: JWT认证,BCrypt密码加密,RBAC权限控制,SQL注入/XSS防护

### 1.2 系统设计与实现概述

#### 1.2.1 系统架构
系统采用**前后端分离架构**,分为三层:

```
┌─────────────────────────────────────────┐
│          前端展示层 (Vue 3)              │
│  - 用户界面组件                          │
│  - 路由管理 (Vue Router)                 │
│  - 状态管理 (Pinia)                      │
│  - API调用 (Axios)                       │
└─────────────────────────────────────────┘
                    ↕ HTTP/HTTPS
┌─────────────────────────────────────────┐
│         后端服务层 (Spring Boot)         │
│  - 控制器层 (Controller)                 │
│  - 业务逻辑层 (Service)                  │
│  - 数据访问层 (Mapper/DAO)               │
│  - 拦截器 (TokenInterceptor)             │
│  - 全局异常处理 (GlobalExceptionHandler) │
└─────────────────────────────────────────┘
                    ↕
┌─────────────────────────────────────────┐
│         数据存储与外部服务               │
│  - MySQL数据库                           │
│  - 阿里云OSS文件存储                     │
│  - Docker评测沙箱                        │
│  - AI大模型API                           │
└─────────────────────────────────────────┘
```

#### 1.2.2 核心模块设计

**1. 认证授权模块**
- JWT Token机制: 登录时签发Token,有效期7天
- Token拦截器: 验证请求头中的Token,解析用户ID和角色
- 权限控制: 基于角色的访问控制(RBAC),区分普通用户(user)和管理员(admin)
- 密码安全: BCrypt加密存储,强度校验(8-20位,包含字母和数字)

**2. 题库管理模块**
- 题目实体(Problem): 标题、描述、难度、分类、时间限制、内存限制、通过率
- 分类实体(Category): 数组、动态规划、贪心、搜索等
- 多对多关联: problem_category表实现题目与分类的灵活关联
- 题解实体(Solution): 题目对应的参考代码和解析

**3. 评测引擎模块**
- 提交实体(Submission): 用户代码、编程语言、评测结果、运行时间、内存占用
- 测试用例(Test Case): 输入文件(.in)、输出文件(.out),存储于OSS
- 评测流程: 代码编译 → 沙箱运行 → 输出比对 → 结果判定
- 评测状态: 0-通过(AC), 1-编译错误(CE), 2-运行错误(RE), 3-超时(TL)
- Docker隔离: 每个评测任务在独立容器中执行,限制CPU、内存、时间资源

**4. AI模块**
- AI题目生成: 根据难度和分类自动生成原创题目,包含完整描述和测试用例
- 代码评价: 分析提交代码的时间复杂度、空间复杂度、代码风格、可读性、优化建议
- 评价实体(AI Evaluation): 存储AI生成的评价报告

**5. 用户统计模块**
- 用户刷题统计(User Statistics): 总提交数、总通过数
- 用户题目状态(User Problem): 记录用户对每道题的通过状态
- 自动更新: 评测完成后触发统计数据更新

#### 1.2.3 数据库设计
核心数据表(共12张):
- `user`: 用户信息表(username, email, password, nickname, avatar, phone, role)
- `user_token`: Token存储表(user_id, token, expire_time)
- `problem`: 题目信息表(title, difficulty, description, time_limit, memory_limit, pass_rate)
- `category`: 分类表(category_name)
- `problem_category`: 题目分类关联表
- `test_case`: 测试用例表(problem_id, input_url, output_url)
- `submission`: 提交记录表(user_id, problem_id, language, code, result, run_time, memory)
- `user_problem`: 用户题目状态表(user_id, problem_id, status)
- `user_statistics`: 用户统计表(user_id, total_submit, total_accept)
- `solution`: 题解表(problem_id, title, content, language, code)
- `ai_evaluation`: AI评价表(submission_id, time_complexity, space_complexity, code_style, readability, optimization)

#### 1.2.4 接口设计
- RESTful API设计,统一路径前缀 `/api/v1`
- 统一响应格式: `{code, message, data}`
- 状态码规范: 200成功, 400参数错误, 401未授权, 403权限不足, 404资源不存在, 500系统异常
- 接口总数: 约60+个接口,覆盖认证、用户、题库、评测、AI、管理员六大模块

#### 1.2.5 安全设计
- **认证安全**: JWT Token验证,Token过期自动失效,退出登录销毁Token
- **数据安全**: BCrypt密码加密,MyBatis预编译防SQL注入,Vue自动转义防XSS
- **文件安全**: 文件类型校验(jpg/jpeg/png/webp),大小限制(10MB),OSS存储
- **评测安全**: Docker沙箱隔离,资源限制(CPU、内存、时间),防止恶意代码
- **权限安全**: 管理员接口角色校验,普通用户无法访问后台管理功能

### 2.3 测试实施

#### 2.3.2 测试用例执行策略

**1. 功能测试执行**
- 按照模块优先级依次执行: 认证 → 用户 → 题库 → 评测 → AI → 管理员
- 每个测试用例执行3遍: 正常场景、异常场景、边界场景
- 记录实际结果,与预期结果对比,标记通过/失败状态
- 失败用例提交缺陷报告,跟踪修复进度

**2. 接口测试执行**
- 使用Postman创建测试集合,覆盖所有API接口
- 验证请求参数、响应格式、状态码、业务逻辑
- 自动化测试脚本: 使用Newman命令行工具批量执行
- 生成测试报告,统计通过率

**3. 性能测试执行**
- 使用JMeter创建测试计划,模拟多用户并发
- 测试场景: 登录(100并发)、题目列表(100并发)、代码提交(50并发)、AI生成(20并发)
- 监控指标: 响应时间、吞吐量、错误率、CPU/内存使用率
- 分析性能瓶颈,提出优化建议

**4. 安全测试执行**
- 手动测试: SQL注入、XSS攻击、文件上传漏洞、权限绕过
- 工具扫描: 使用OWASP ZAP进行自动化安全扫描
- 验证防护机制: Token验证、密码加密、沙箱隔离、资源限制

**5. 前端测试执行**
- 浏览器兼容性测试: Chrome、Firefox、Safari、Edge
- 响应式设计测试: 桌面端(1920x1080)、平板(768x1024)、手机(375x667)
- 用户交互测试: 表单验证、路由跳转、状态管理、组件渲染
- 自动化测试: 使用Cypress编写E2E测试脚本

#### 2.3.4 缺陷管理流程

**1. 缺陷发现**
- 测试执行过程中发现不符合预期的行为
- 记录缺陷现象、复现步骤、环境信息、截图/日志

**2. 缺陷报告**
使用标准缺陷报告模板:
```
缺陷ID: BUG-001
模块: 认证管理
严重程度: 高/中/低
优先级: 紧急/高/中/低
标题: 用户注册时邮箱唯一性校验失效
复现步骤:
  1. 使用邮箱test@example.com注册用户A,注册成功
  2. 使用相同邮箱test@example.com注册用户B
  3. 预期结果: 返回"该邮箱已被注册"
  4. 实际结果: 注册成功,数据库出现重复邮箱记录
环境: Windows 11, Chrome 120, JDK 17
附件: 截图、日志文件
```

**3. 缺陷修复**
- 开发人员接收缺陷报告,分析根因
- 修复代码,编写单元测试验证
- 提交代码,标注关联的缺陷ID

**4. 缺陷验证**
- 测试人员重新执行相关测试用例
- 验证缺陷是否修复,是否引入新问题
- 更新缺陷状态: 已修复/重新打开

**5. 缺陷统计**
- 每日统计新增缺陷、已修复缺陷、待修复缺陷
- 每周生成缺陷趋势报告
- 测试结束时输出缺陷分析总结

### 2.4 通过准则

#### 2.4.1 功能测试通过准则

| 测试类型 | 通过标准 | 指标要求 |
|---------|---------|---------|
| 单元测试 | 核心业务逻辑覆盖率≥70%,所有用例通过 | 通过率100% |
| 接口测试 | 所有API接口功能正常,响应格式正确 | 通过率≥98% |
| 集成测试 | 模块间交互正常,数据流转无误 | 通过率100% |
| 系统测试 | 完整业务流程验证通过,前端界面正常 | 通过率≥95% |
