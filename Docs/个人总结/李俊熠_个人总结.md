## 一、角色与职责概述

在本项目中，我担任**项目组长**，负责项目的整体统筹与全流程管控。工作横跨**项目管理、文档体系建设、技术选型决策与核心模块开发**四条主线：对内组织团队分工协作、把控里程碑节点与进度风险，对外牵头确立技术方向、主导核心文档编写，同时承担在线代码编辑器与用户管理模块的开发工作。整体目标是保障「需求分析 → 系统设计 → 编码开发 → 联调测试」全流程平稳推进。

## 二、主要工作内容

### 1. 项目管理与团队统筹（贯穿全程）
- 组织召开**项目启动会**，明确项目核心定位、团队扁平化管理机制与协作规范；
- 制定**12 周里程碑计划**，完成整体预算编制与资源分配方案，搭建项目全周期推进框架；
- 完善代码管理、文档管理、版本管控等标准化协作规范，开展**团队开发规范专项培训**，并形成《开发规范常见问题手册》；
- 搭建**可视化进度看板**与**分级风险预警台账**，建立周度进度核对与风险上报机制，推动进度管控从「口头同步」向「可视化、可追溯」转型。

### 2. 技术选型与架构决策
- 牵头开展技术栈调研与选型，最终确立 **Spring Boot 2.7.x + Vue + MySQL + C++ 本地编译环境**的核心技术栈；
- 主持系统架构设计专项评审会，确认 **「传统判题 + AI 赋能」双引擎**核心数据流向，对齐团队架构共识。

### 3. 核心文档体系建设
- 牵头编制《立项计划书》，完成项目正式立项；
- 统筹主持并独立撰写《需求分析文档（SRS）》中项目概述、验收标准等核心模块，完成全文档意见收集、多轮迭代与最终定稿；
- 牵头编写《软件详细设计文档（SDD）》，覆盖系统架构、模块设计、数据库、接口规范等；采用**「线上预评审 + 线下集中评审」**模式完成全流程评审，闭环所有评审意见并正式定稿发布，作为后续开发、测试、运维的唯一基准依据。

### 4. 环境搭建与核心模块开发
- 完成项目**沙箱判题环境**的基础搭建（服务器配置、依赖部署、运行环境调试与可用性验证），并完成单节点性能基准测试，输出测试报告、定位瓶颈；
- 完成**在线代码编辑器**基础框架搭建、接口定义、核心模块编码及单元测试；
- 完成**用户管理模块**表结构设计，并启动其接口开发；
- 牵头制定跨模块联调方案，明确节点、责任人与验收标准。

## 三、遇到的问题与解决方法

| 问题 | 解决思路 |
| --- | --- |
| 文档串行编写（SRS→SDD）耗时过长，拖慢进度 | 拆分编写任务、分模块编制分阶段评审，并在定稿阶段同步启动下一阶段筹备工作 |
| 第三方 AI 大模型 API 输出格式不稳定 | 建立核心技术对接风险预判机制，提前开展多场景测试并预设应对方案 |
| 沙箱环境高可用滞后（云资源审批延迟、负载均衡未落地） | 安排专人跟进资源审批、申请临时备用资源，优先评审负载均衡方案，资源到位即启动部署压测 |
| 进度依赖口头同步、缺乏可视化跟踪 | 搭建并强制使用进度看板，建立风险预警台账与每日早会核对机制 |
| 部分成员单元测试覆盖率低、文档/注释不规范 | 将单元测试覆盖率纳入代码评审强制指标，开展专项培训并一对一指导薄弱成员 |
| 跨模块接口定义存在歧义 | 要求各模块负责人每日同步接口变更、提前对齐接口文档 |

## 四、收获与反思

1. **项目管理能力**：从立项、里程碑规划到进度、风险、预算管控，完整经历了一个软件项目的全生命周期管理，深刻体会到「前置风险研判」与「可视化进度管控」对项目按期交付的关键作用。
2. **文档驱动开发**：主导 SRS / SDD 两大核心文档的编写与评审，认识到高质量、经评审定稿的设计文档是减少后续开发返工、统一团队认知的基础。
3. **技术与工程结合**：在双引擎架构选型、沙箱环境搭建、编辑器与用户管理模块开发中，锻炼了将架构决策落地为可运行系统的工程能力。
4. **待改进之处**：文档评审与开发存在一定串行耦合，导致前期进度偏紧；后续应更早推动设计、开发、测试并行筹备，并将单元测试覆盖率、接口对齐等规范在开发初期就刚性落地，减少后期联调成本。
